# 毕设：文档与日志体系（模板）

这套模板的目标是：
1) **把每次讨论确认的结论（定稿）追加到日志**，形成可追溯记录；
2) **把“定稿结论”转成 Codex 可直接执行的工单**（接口/表结构/验收标准明确）。

## 建议的文件分工（最小集合）
- `定稿记录.txt`：开题阶段确定的“范围/功能清单/里程碑”基线（尽量少改动，作为需求冻结参考）
- `log记录/决策日志.md`：只记录**已确认**的决定（Append-only）
- `log记录/开发日志.md`：按日期追加开发进展与结论（Append-only）
- `log记录/Codex任务单.md`：每次定稿后生成的可执行任务（Append-only，可维护状态）
- `docs/项目蓝图.md`：1页总览（范围、闭环、模块地图）
- `docs/开发计划.md`：从现在起按周滚动维护
- `docs/技术框架书.md`：架构与技术路线（落到模块、数据流、关键组件）
- `docs/接口与数据库.md`：核心表结构 + API 清单（可后续拆分为 OpenAPI + 数据字典）
- `docs/部署说明.md`：本地/演示环境如何跑起来（步骤可复现）
- `docs/测试用例.md`：关键路径用例（登录/RBAC、报告提交、签到、审批、批阅导出）
- `docs/开发规范.md`：分支/提交信息/命名/代码结构/日志要求

> 原则：先少后多。模板先保证“能推进开发 + 能写论文证据链”，后面再按需要补充。
